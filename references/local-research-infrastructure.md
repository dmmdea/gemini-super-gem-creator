# Local Research Infrastructure Reference

Local Gem replaces NotebookLM notebooks with local source directories and an optional progressive research stack. This reference covers the complete infrastructure: source directory architecture, research tool setup, RAG pipeline configuration, and source directory management.

---

## Source Directory Architecture

### Directory Structure

```
[Gem Memory Folder]/
├── _Prompt_[Name]_GemSystemPrompt_v1.0.md
├── _Memory_[Name]_GemMemoryHub_Active.md
├── _Memory_[Name]_GemMemoryHub_Archive.md
├── _Progress_[Name]_ActiveTask.md
├── _Heuristics_[Name]_Pool.md
├── sources/
│   ├── core/           # Equivalent to NB1: fundamentals, current state
│   │   ├── doc1.md
│   │   ├── doc2.pdf
│   │   └── .index      # RAG index (Chroma, auto-generated by --index)
│   ├── expansion/      # Equivalent to NB2: growth, alternatives
│   │   ├── doc1.md
│   │   └── .index
│   ├── [domain-3]/     # Additional partitions (no limit)
│   ├── [domain-4]/
│   └── [domain-5]/
└── research-cache/     # Cached SearXNG/Perplexica results
    ├── 2026-04-07_topic-slug.md
    └── 2026-04-08_topic-slug.md
```

### Key Differences from NotebookLM

| Aspect | NotebookLM (Cloud Tiers) | Source Directories (Local Gem) |
|--------|--------------------------|-------------------------------|
| Container | Notebook (max 2 Standard / 5 Research) | Directory (unlimited) |
| Source format | Uploaded docs + Deep Research results | Any local file (.md, .pdf, .txt, .html, .csv) |
| Source limit | 50–300 per notebook (tier-dependent) | No hard limit — practical limit is index size |
| Retrieval | NotebookLM's built-in grounding | RAG via Chroma + Ollama embeddings (Layer 2) or raw file reading (Layer 0) |
| Research | NotebookLM Deep Research (cloud) | SearXNG raw search (Layer 1) or Perplexica AI-synthesized (Layer 3) |
| Privacy | Google servers | Fully local |
| Cost | Free (consumer tier) | Free (local compute) |

### Source Directory Naming Convention

Follow the same semantic naming principles as NotebookLM notebooks:

- `core/` — Fundamentals, current state, primary domain knowledge (always created)
- `expansion/` — Growth, alternatives, future planning (always created)
- `[domain-specific]/` — Additional partitions named by sub-domain (created as needed)

The skill generates directory names during the research plan phase. Names should be lowercase, hyphenated, and descriptive: `regulatory-compliance/`, `market-analysis/`, `technical-specs/`.

### Supported Source File Types

The gem script can read the following file types from source directories:

| Type | Extension | How Processed |
|------|-----------|---------------|
| Markdown | `.md` | Direct text injection or RAG-indexed |
| Plain text | `.txt` | Direct text injection or RAG-indexed |
| PDF | `.pdf` | Text extraction via Python (pdfplumber, optional dep) or raw path reference |
| HTML | `.html` | Text extraction (strip tags) or RAG-indexed |
| CSV | `.csv` | Header + first N rows as context, or RAG-indexed |
| JSON | `.json` | Pretty-printed excerpt or RAG-indexed |

**Unsupported types** (images, video, audio, binary) are skipped silently. The script logs skipped files at `--verbose` level.

## Progressive Research Stack

### Design Philosophy

Research infrastructure is 100% optional at initial setup. The minimum viable Local Gem works with manual document curation (user drops files in a folder). Each layer builds progressively — no layer requires any other layer except where noted.

### Layer 0: Manual Curation (DEFAULT — Zero Setup)

User manually adds documents to `sources/core/` and `sources/expansion/`. The gem script reads them as raw context — no indexing, no Docker, no dependencies.

**How it works:** On each conversation turn, the daemon reads all files in the active source directories and injects relevant content as context alongside the memory hub. Without RAG indexing (Layer 2), the script uses a simple strategy: inject the first N characters from each source file, prioritizing files by recency and directory relevance.

**When this is enough:** Gems with a small, stable knowledge base (< 20 documents total). The user curates sources manually and updates them as needed.

**Limitations:** Without RAG, context injection is limited by the model's context window. Large source collections (50+ documents) will exceed context limits — upgrade to Layer 2 for semantic retrieval.

### Layer 1: SearXNG Web Search (OPT-IN — One Docker Command)

SearXNG is a self-hosted meta-search engine that aggregates results from 90+ search providers (Google, Bing, DuckDuckGo, etc.) without tracking or ads.

```bash
# Install and run SearXNG (one command)
docker run -d --name searxng -p 8888:8888 searxng/searxng

# Verify it's running
curl -s http://localhost:8888/search?q=test&format=json | head -c 200

# Script auto-detects SearXNG at localhost:8888 on next startup
```

**How the gem script uses SearXNG:**
1. Model outputs `[RESEARCH_NEEDED: topic]` during conversation
2. Daemon intercepts the tag and sends the query to SearXNG's JSON API
3. SearXNG returns top 10 results (title, URL, snippet)
4. Daemon injects results as context for the model to synthesize
5. Raw results are cached in `research-cache/` with timestamp

**Privacy:** SearXNG runs locally. Search queries never leave your network. Results are fetched from public search engines but SearXNG strips tracking parameters and rotates across providers.

**Configuration (optional, in gem script config):**
```python
"searxng_url": "http://localhost:8888",  # Default, auto-detected
"searxng_engines": "google,bing,duckduckgo",  # Default engine set
"searxng_max_results": 10,  # Results per query
```

### Layer 2: RAG Indexing (OPT-IN — One Script Flag)

Built-in RAG (Retrieval-Augmented Generation) enables semantic search over local documents. Instead of injecting entire files as context, the script indexes documents into vector embeddings and retrieves only the most relevant chunks per query.

```bash
# Index all source directories (one command)
python gem-script.py --index

# The script auto-pulls nomic-embed-text via Ollama on first run
# Uses Chroma for vector storage (pip install chromadb — optional, gracefully skipped)
```

**How it works:**
1. `--index` scans all `sources/*/` subdirectories
2. Each document is split into chunks (512 tokens default)
3. Chunks are embedded using `nomic-embed-text` via Ollama's embedding API
4. Embeddings are stored in Chroma (per-directory `.index` collections)
5. On each conversation turn, the daemon embeds the user's query and retrieves top-K chunks
6. Retrieved chunks are injected as grounding context alongside the memory hub

**Dependencies:**
- `chromadb` (Python): `pip install chromadb` — optional. If not installed, RAG features degrade gracefully to Layer 0 raw file reading. The script prints: `💡 Want semantic search? Run: pip install chromadb`
- `nomic-embed-text` (Ollama): Auto-pulled on first `--index` run. No user action needed.

**RAG Configuration (in gem script config):**
```python
"rag_enabled": "auto",              # True/False/"auto" — auto enables if sources/ exists + chromadb installed
"source_dirs": "auto",              # "auto" scans sources/*/ subdirectories
"rag_top_k": 5,                     # Chunks retrieved per query
"rag_chunk_size": 512,              # Tokens per chunk
"rag_embedding_model": "nomic-embed-text",  # Embedding model via Ollama
```

**Index management commands:**
```bash
python gem-script.py --index          # Rebuild all indexes
python gem-script.py --index core     # Rebuild only the 'core' directory index
python gem-script.py --stats          # Show index stats (doc count, chunk count, last indexed)
python gem-script.py --add-source file.md core  # Add a file to core/ and re-index
```

**When to upgrade from Layer 0 to Layer 2:**
- Source collection exceeds ~20 documents or ~50K words total
- User needs precise retrieval ("what did document X say about Y?")
- Multiple source directories with distinct topics
- Context window pressure from large source collections

### Layer 3: Perplexica AI-Synthesized Research (OPT-IN — One Docker Compose)

Perplexica is a fully local alternative to Perplexity AI. It connects to SearXNG (Layer 1) for web search and Ollama (already running for inference) for AI synthesis. The result: research queries return synthesized, cited answers instead of raw search snippets.

**Prerequisites:** Layer 1 (SearXNG) must be running. Ollama must be running (it already is — it's the inference engine).

**Setup:**
The gem script generates a pre-configured `perplexica-compose.yml` during `--setup` if SearXNG and Ollama are detected:

```yaml
# perplexica-compose.yml (auto-generated by gem-script.py --setup)
services:
  perplexica-backend:
    image: itzcrazykns1337/perplexica-backend:main
    ports:
      - "3001:3001"
    environment:
      - SEARXNG_URL=http://host.docker.internal:8888  # SearXNG from Layer 1
      - OLLAMA_URL=http://host.docker.internal:11434   # Ollama (inference engine)
      - DEFAULT_MODEL=gemma4:e4b                       # Uses your gem's model
    extra_hosts:
      - "host.docker.internal:host-gateway"
    restart: unless-stopped

  perplexica-frontend:
    image: itzcrazykns1337/perplexica-frontend:main
    ports:
      - "3002:3000"
    depends_on:
      - perplexica-backend
    restart: unless-stopped
```

```bash
# Start Perplexica (one command — after SearXNG is running)
docker compose -f perplexica-compose.yml up -d

# Script auto-detects Perplexica at localhost:3001 on next startup
```

**How the gem script uses Perplexica:**
1. Model outputs `[RESEARCH_NEEDED: topic]` during conversation
2. Daemon sends the research query to Perplexica's API (`localhost:3001`)
3. Perplexica runs multiple web searches via SearXNG, then synthesizes results using the local Gemma model
4. Returns a coherent answer with source citations
5. Daemon auto-injects the synthesized answer as context and caches it in `research-cache/`
6. Optionally adds the result to the relevant source directory for future RAG retrieval

**Why this matters:** This is where Local Gem genuinely surpasses Research Gem. NotebookLM Deep Research is powerful but it's a black box on Google's servers. Perplexica gives the same deep research capability, fully local, with your own model doing the synthesis, and results feed directly into source directories.

**Research routing priority (automatic):**

| Research backend detected | Routing behavior | Quality |
|--------------------------|-----------------|---------|
| Perplexica (Layer 3) | Auto-inject synthesized answer with citations → cache in `research-cache/` | High — pre-synthesized, grounded |
| SearXNG only (Layer 1) | Return raw search snippets → model synthesizes in-turn | Medium — raw results need model judgment |
| No research infra | Print: "Please research this manually and add results to `sources/`" | User-dependent |

**Configuration (in gem script config):**
```python
"perplexica_enabled": "auto",       # True/False/"auto" — auto-detects at localhost:3001
"perplexica_url": "http://localhost:3001",
```

## Research Cache Management

All research results (from SearXNG or Perplexica) are cached in the `research-cache/` directory:

```
research-cache/
├── 2026-04-07_market-analysis-saas-pricing.md
├── 2026-04-07_competitor-feature-comparison.md
└── 2026-04-08_regulatory-update-gdpr.md
```

**Cache file format:**
```markdown
# Research: [query]
**Source:** [SearXNG | Perplexica]
**Date:** [timestamp]
**Query:** [original research query]

## Results
[Raw snippets (SearXNG) or synthesized answer (Perplexica)]

## Sources
- [URL 1] — [title]
- [URL 2] — [title]
```

**Cache commands:**
```bash
python gem-script.py --stats          # Shows research cache size and entry count
python gem-script.py --clear-cache    # Clears research cache (source dirs unaffected)
```

**Auto-promotion:** When Perplexica returns a high-quality synthesized answer, the daemon can optionally copy it into the relevant source directory for permanent RAG indexing. This is controlled by the config flag `"research_auto_promote": True` (default: True for Perplexica results, False for raw SearXNG).

## Source Directory Management Commands

The gem script includes built-in source directory management:

```bash
# View source directory stats
python gem-script.py --stats
# Output:
#   sources/core/        — 12 documents, 45K words, indexed 2026-04-07
#   sources/expansion/   — 8 documents, 28K words, indexed 2026-04-07
#   research-cache/      — 5 entries, last query 2026-04-07

# Add a file to a source directory and re-index
python gem-script.py --add-source report.pdf core
# Copies report.pdf to sources/core/, re-indexes if RAG enabled

# Rebuild all indexes
python gem-script.py --index

# Rebuild a specific directory's index
python gem-script.py --index expansion

# Create a new source directory
python gem-script.py --new-source regulatory-compliance
# Creates sources/regulatory-compliance/ with empty .index
```

## Health Monitoring (Research-Specific)

The gem script's health monitoring includes research infrastructure checks:

| Check | Trigger | Message |
|-------|---------|---------|
| Empty source directory | `sources/core/` has 0 files | `⚠️ sources/core/ is empty. Add documents: python gem-script.py --add-source file.md core` |
| Stale RAG index | Index older than 30 days | `💡 RAG index is stale. Rebuild: python gem-script.py --index` |
| SearXNG unreachable | Layer 1 was available, now isn't | `⚠️ SearXNG not responding at localhost:8888. Research queries will use fallback.` |
| Perplexica unreachable | Layer 3 was available, now isn't | `⚠️ Perplexica not responding at localhost:3001. Research queries will use SearXNG fallback.` |
| Research cache large | Cache exceeds 100 entries | `💡 Research cache has [N] entries. Review or clear: python gem-script.py --clear-cache` |

Every warning includes the exact command to fix it. No warning is ever just "something is wrong."

## Integration with Gem Script

The research infrastructure is built into the gem script (`local-gem-script-template.py`), not a separate tool. Key integration points:

1. **Auto-detection on startup:** Script probes `localhost:8888` (SearXNG) and `localhost:3001` (Perplexica) — uses what's available, skips what isn't
2. **Source context injection:** On each turn, daemon injects relevant source content (raw files or RAG-retrieved chunks) alongside the memory hub
3. **Research routing:** `[RESEARCH_NEEDED: topic]` tags in model output are intercepted and routed to the best available backend
4. **Cache management:** Research results are cached and optionally promoted to source directories
5. **Index management:** `--index`, `--stats`, `--add-source` commands are built into the script
6. **Health checks:** Research infra health is checked on daemon start and periodically during operation

**The user never installs a separate RAG pipeline, search aggregator config, or research tool.** Everything is either built into the gem script or added with a single Docker command.
