# Notebook Theme Examples by Domain

## Theme Selection Guidelines

### Standard Gem (2 Notebooks)

1. **NB1 = what exists now.** Current operations, known constraints, fundamentals, seed documents. What the gem needs to ground its day-to-day advice.
2. **NB2 = what could be.** Future planning, optimization, alternatives, expansion. What the gem needs to give growth-oriented advice.
3. **Keep themes independent enough.** Each notebook should make sense on its own. Avoid overlap that splits the same topic across both notebooks.
4. **Match the user's biggest challenge.** If they said "my biggest problem is X," make sure X is prominent in one of the two notebooks — don't bury it.
5. **Never plan more than 2.** If the domain feels like it needs 3+ notebooks, consolidate. The knowledge slot budget is non-negotiable.

### Research Gem (5 Notebooks)

1. **Partition by sub-domain, not by time horizon.** Unlike Standard's now/future split, Research notebooks each cover a distinct knowledge area.
2. **NB1 = foundations the others build on.** Core principles, established best practices, first principles. The "textbook" content.
3. **NB2 = how-to.** Technical implementation, tools, hands-on methodology. The "practitioner" content.
4. **NB3 = alternatives and trade-offs.** Competing approaches, vendor comparisons, decision frameworks. The "analyst" content.
5. **NB4 = cutting edge.** Emerging research, novel approaches, future directions. The "researcher" content.
6. **NB5 = evidence.** Benchmarks, case studies, performance data, real-world outcomes. The "evidence base" content.
7. **Keep notebooks independent.** Each should make sense if read alone. No notebook should require reading another to be useful.
8. **Never plan more than 5.** If the domain seems to need 6+, consolidate. 5 notebooks already consume 5 of ~10 Knowledge slots.
9. **Remember the memory trade-off.** Research Gems have a leaner memory hub (5–6K target / 10K ceiling) because notebooks consume more Knowledge context. Design notebook themes to minimize the need for memory hub compensations.

### Local Gem (Source Directories — Unlimited)

1. **Start with `core/` + `expansion/` (same as Standard's NB1/NB2).** These two directories are always created. They follow the same now/future split as Standard Gem notebooks.
2. **Add domain-specific directories as needed.** Unlike cloud tiers, there's no slot limit. If the domain naturally partitions into 4 or 7 areas, create that many directories.
3. **Name directories descriptively.** Lowercase, hyphenated: `regulatory-compliance/`, `market-analysis/`, `technical-specs/`. The name is the only metadata — make it count.
4. **Each directory = one topic.** Same independence rule as Research Gem notebooks. Each directory should make sense on its own.
5. **Supported formats:** `.md`, `.txt`, `.pdf`, `.html`, `.csv`, `.json`. Unsupported types are silently skipped.
6. **No source count limit per directory.** Practical limit is RAG index size and retrieval latency. For directories with 50+ documents, RAG indexing (Layer 2) is strongly recommended.
7. **Research cache is separate.** `research-cache/` stores SearXNG/Perplexica results. These are not source directories — they're transient query results that can optionally be promoted to a source directory for permanent indexing.
